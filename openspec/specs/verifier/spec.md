# Verifier Specification

> Generated by spec-gen v1.0.0 on 2026-03-01 16:52
> Source files: src/core/verifier/verification-engine.ts

## Purpose

Verifies whether generated OpenSpec specifications accurately describe the codebase by predicting
file behavior from specs and comparing against actual files.

## Entities

### VerificationCandidate

Represents a source code file selected for verification against OpenSpec specifications. It includes
metadata about the file's complexity and usage to determine its suitability for verification.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| path | string | Relative path of the file within the project. |
| absolutePath | string | Absolute filesystem path of the file. |
| domain | string | Business domain this file belongs to, derived from the project structure. |
| usedInGeneration | boolean | Flag indicating whether this file was used in the generation of the specifications (excluded from verification if true). |
| complexity | number | Cyclomatic complexity or other metric representing the file's complexity. |
| lines | number | Number of lines in the file. |
| imports | number | Number of import statements in the file. |
| exports | number | Number of export statements in the file. |

**Relationships:**

- has one VerificationResult (A VerificationCandidate produces one VerificationResult after verification.)

### FilePrediction

Represents the LLM's prediction of a file's purpose, structure, and logic based solely on OpenSpec
specifications. Used to compare against actual file content.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| predictedPurpose | string | Predicted high-level purpose of the file based on specifications. |
| predictedImports | string[] | List of predicted import statements the file should contain. |
| predictedExports | string[] | List of predicted export statements the file should contain. |
| predictedLogic | string[] | List of key logic patterns expected in the file based on specifications. |
| relatedRequirements | string[] | List of requirement identifiers from the specifications that this file should implement. |
| confidence | number | Confidence score (0-1) indicating how certain the LLM is about this prediction. |
| reasoning | string | Explanation of how the prediction was derived from the specifications. |

**Relationships:**

- has one VerificationResult (A FilePrediction is compared against actual file content to produce a VerificationResult.)

### PurposeMatch

Represents the comparison result between the predicted and actual purpose of a file. Measures
semantic similarity between the two descriptions.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| predicted | string | Predicted purpose of the file from FilePrediction. |
| actual | string | Actual purpose of the file derived from code analysis or documentation. |
| similarity | number | Semantic similarity score (0-1) between predicted and actual purpose. |

**Relationships:**

- has one VerificationResult (A PurposeMatch is a component of a VerificationResult.)

### SetMatch

Represents the comparison result between predicted and actual sets of imports or exports. Measures
precision, recall, and F1 score of the prediction.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| predicted | string[] | Predicted set of imports or exports from FilePrediction. |
| actual | string[] | Actual set of imports or exports extracted from the file. |
| precision | number | Precision score (0-1) indicating how many predicted items were correct. |
| recall | number | Recall score (0-1) indicating how many actual items were predicted. |
| f1Score | number | F1 score (0-1) representing the harmonic mean of precision and recall. |

**Relationships:**

- has one VerificationResult (A SetMatch is a component of a VerificationResult.)

### VerificationResult

Represents the outcome of verifying a source code file against its predicted specification. Contains
detailed comparison metrics and an overall verification score.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| candidate | VerificationCandidate | The file that was verified. |
| prediction | FilePrediction | The LLM's prediction of the file's content based on specifications. |
| purposeMatch | PurposeMatch | Comparison result between predicted and actual file purpose. |
| importsMatch | SetMatch | Comparison result between predicted and actual imports. |
| exportsMatch | SetMatch | Comparison result between predicted and actual exports. |
| logicMatch | number | Semantic similarity score (0-1) between predicted and actual file logic. |
| overallScore | number | Weighted average score (0-1) representing the overall verification result. |
| verificationPassed | boolean | Flag indicating whether the file passed verification based on the overall score and threshold. |
| issues | string[] | List of issues or discrepancies found during verification. |

**Relationships:**

- has one VerificationCandidate (A VerificationResult is produced from one VerificationCandidate.)
- has one FilePrediction (A VerificationResult is based on one FilePrediction.)

### RequirementCoverage

Represents the coverage of requirements in a file, indicating which requirements are implemented and
the overall coverage score.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| relatedRequirements | string[] | List of requirements that the file is expected to implement. |
| actuallyImplements | string[] | List of requirements that are actually implemented in the file. |
| coverage | number | Coverage score (0-1) indicating the proportion of requirements implemented. |

**Relationships:**

- many-to-one FilePrediction (A RequirementCoverage result is derived from comparing a FilePrediction's related requirements to the actual requirements implemented in a VerificationCandidate.)

### VerificationReport

Represents the outcome of verifying a set of specifications against a codebase. It contains
aggregated metrics, domain-specific breakdowns, and actionable recommendations for improving the
specs.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| timestamp | string | The date and time when the verification report was generated. |
| specVersion | string | The version of the specifications being verified. |
| sampledFiles | number | The total number of files sampled and verified in the codebase. |
| passedFiles | number | The number of files that passed verification based on the confidence threshold. |
| overallConfidence | number | The aggregated confidence score (0 to 1) representing how well the specs describe the codebase overall. |
| recommendation | string | A high-level recommendation based on the verification results. Possible values: 'ready', 'needs-review', or 'regenerate'. |
| domainBreakdown | array | A breakdown of verification results grouped by domain, including scores and weakest areas. |
| commonGaps | array | A list of recurring issues or gaps identified across the codebase during verification. |
| suggestedImprovements | array | A list of actionable suggestions for improving the specs, grouped by domain. |
| results | array | Detailed verification results for each file, including scores and feedback. |

**Relationships:**

- contains DomainBreakdown (The VerificationReport contains one or more DomainBreakdown entries, each representing a specific domain's verification results.)
- contains FileVerificationResult (The VerificationReport contains one or more FileVerificationResult entries, each representing the verification outcome for a specific file.)
- contains SuggestedImprovement (The VerificationReport may contain zero or more SuggestedImprovement entries, providing actionable feedback for improving the specs.)

### DomainBreakdown

Represents the verification results for a specific domain within the codebase, including aggregated
scores and weakest areas.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| domain | string | The name of the domain being verified (e.g., 'User Management', 'Authentication'). |
| specPath | string | The file path to the specification file for this domain. |
| filesVerified | number | The number of files verified for this domain. |
| averageScore | number | The average confidence score (0 to 1) for files in this domain. |
| weakestArea | string | A description of the weakest aspect of the specs for this domain (e.g., 'Missing export definitions'). |

### FileVerificationResult

Represents the verification outcome for a single file, including scores across multiple categories
and feedback.

**Properties:**

| Name | Type | Description |
|------|------|-------------|
| filePath | string | The path to the file that was verified. |
| domain | string | The domain to which this file belongs. |
| overallScore | number | The aggregated confidence score (0 to 1) for this file. |
| llmConfidence | number | The confidence score (0 to 1) provided by the LLM for this file's verification. |
| purposeMatch | object | The similarity score (0 to 1) indicating how well the file's purpose matches the spec. |
| importMatch | object | The F1 score (0 to 1) indicating how well the file's imports match the spec. |
| exportMatch | object | The F1 score (0 to 1) indicating how well the file's exports match the spec. |
| requirementCoverage | object | The coverage score (0 to 1) indicating how well the file's requirements are covered by the spec. |
| feedback | array | A list of specific feedback items for improving the specs related to this file. |

## Requirements

### Requirement: VerificationCandidateValidation

The system SHALL validate VerificationCandidate according to these rules:
- path must be a non-empty string
- absolutePath must be a valid filesystem path
- domain must be a non-empty string
- complexity must be a positive integer
- lines must be a positive integer
- imports must be a non-negative integer
- exports must be a non-negative integer

#### Scenario: ExcludeFilesUsedInGeneration
- **GIVEN** A list of VerificationCandidates and a list of files used in specification generation
- **WHEN** Selecting candidates for verification
- **THEN** Files marked as usedInGeneration should be excluded from the verification process

#### Scenario: FilterByComplexity
- **GIVEN** A list of VerificationCandidates with varying complexity values and engine options specifying
min/max complexity
- **WHEN** Selecting candidates for verification
- **THEN** Only files with complexity within the specified range should be selected

### Requirement: FilePredictionValidation

The system SHALL validate FilePrediction according to these rules:
- predictedPurpose must be a non-empty string
- predictedImports must be an array of strings
- predictedExports must be an array of strings
- predictedLogic must be an array of strings

- confidence must be a number between 0 and 1
- reasoning must be a non-empty string

#### Scenario: HighConfidencePrediction
- **GIVEN** A FilePrediction with confidence >= 0.8 and well-defined specifications
- **WHEN** Comparing prediction against actual file content
- **THEN** The prediction should closely match the actual file's purpose, imports, and exports

#### Scenario: LowConfidencePrediction
- **GIVEN** A FilePrediction with confidence < 0.5 or vague specifications
- **WHEN** Comparing prediction against actual file content
- **THEN** The prediction may not match the actual file, indicating gaps in the specifications

### Requirement: PurposeMatchValidation

#### Scenario: MismatchedPurpose

### Requirement: SetMatchValidation

The system SHALL validate SetMatch according to these rules:
- predicted must be an array of strings

- **WHEN** Calculating precision, recall, and F1 score

#### Scenario: PartialSetMatch

#### Scenario: FailedVerification
